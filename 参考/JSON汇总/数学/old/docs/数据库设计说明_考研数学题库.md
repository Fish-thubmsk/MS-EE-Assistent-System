# 考研数学题库数据库设计说明（Final 版数据）

## 1. 文档目标

本说明用于将当前的三个最终 JSON 文件：

- `kmath_questions1_final.json`（数学一）
- `kmath_questions2_final.json`（数学二）
- `kmath_questions3_final.json`（数学三）

稳定、规范地落库到关系型数据库（推荐 PostgreSQL），并支持后续检索、统计、导出与训练数据生产。

---

## 2. 数据源现状与约束

### 2.1 当前 JSON（final）结构

每个文件的顶层字段：

- `site`：固定为 `kmath.cn`
- `subject_type`：`math_1 | math_2 | math_3`
- `collection_id`：来源集合 ID
- `papers`：试卷数组

每个 `paper`：

- `subject_type`
- `paper_title`
- `questions`

每个 `question`：

- `question_id`：已重编号，按单卷从 `1..N`
- `question_type`：`single_choice | fill_blank | subjective`
- `stem`：题干（LaTeX 文本可能存在）
- `options`：可选对象（仅选择题存在，键通常是 A-D）

### 2.2 明确删除字段（已删除）

以下字段已经在 final 版去除：

- `paper_id`
- `crawled_at`
- `source_question_id`

这意味着数据库设计要以“业务使用优先、轻量”为目标，不依赖来源系统主键。

---

## 3. 设计原则

1. **科目必须强区分**：数学一/二/三必须可独立查询、统计。
2. **规范化优先**：选项拆表，避免单字段存 JSON 文本导致检索困难。
3. **可追溯卷内顺序**：保留 `question_no`（对应 JSON `question_id`）。
4. **约束先行**：通过 `CHECK/UNIQUE/FK` 保证数据质量。
5. **查询友好**：对高频维度（科目、题型）建索引。

---

## 4. 逻辑模型（ER 概念）

- `subjects`（科目维表）1 --- N `papers`（试卷）
- `papers` 1 --- N `questions`（题目）
- `questions` 1 --- N `question_options`（选项）

---

## 5. 物理模型（PostgreSQL DDL）

> 说明：以下 SQL 可直接执行。

```sql
-- =========================
-- 1) 科目维表
-- =========================
CREATE TABLE subjects (
  subject_code TEXT PRIMARY KEY
    CHECK (subject_code IN ('math_1', 'math_2', 'math_3')),
  subject_name TEXT NOT NULL UNIQUE
);

INSERT INTO subjects(subject_code, subject_name) VALUES
('math_1', '数学一'),
('math_2', '数学二'),
('math_3', '数学三');


-- =========================
-- 2) 试卷表
-- =========================
CREATE TABLE papers (
  id BIGSERIAL PRIMARY KEY,
  subject_code TEXT NOT NULL REFERENCES subjects(subject_code),

  -- 每个科目内的顺序号（导入时按JSON顺序生成：1,2,3...）
  paper_no INTEGER NOT NULL CHECK (paper_no > 0),

  paper_title TEXT NOT NULL,

  -- 同一科目下卷序号唯一
  UNIQUE (subject_code, paper_no),

  -- 同一科目下标题唯一（避免重复导入）
  UNIQUE (subject_code, paper_title)
);


-- =========================
-- 3) 题目表
-- =========================
CREATE TABLE questions (
  id BIGSERIAL PRIMARY KEY,
  paper_id BIGINT NOT NULL REFERENCES papers(id) ON DELETE CASCADE,

  -- 卷内题号：对应JSON question_id（字符串数字，入库后转INT）
  question_no INTEGER NOT NULL CHECK (question_no > 0),

  question_type TEXT NOT NULL
    CHECK (question_type IN ('single_choice', 'fill_blank', 'subjective')),

  stem TEXT NOT NULL,

  -- 同一套卷内题号唯一
  UNIQUE (paper_id, question_no)
);


-- =========================
-- 4) 选项表（仅选择题会有）
-- =========================
CREATE TABLE question_options (
  id BIGSERIAL PRIMARY KEY,
  question_id BIGINT NOT NULL REFERENCES questions(id) ON DELETE CASCADE,

  option_key CHAR(1) NOT NULL
    CHECK (option_key IN ('A', 'B', 'C', 'D', 'E', 'F')),

  option_text TEXT NOT NULL,

  UNIQUE (question_id, option_key)
);


-- =========================
-- 5) 索引（按常用查询场景）
-- =========================
CREATE INDEX idx_papers_subject ON papers(subject_code);
CREATE INDEX idx_questions_paper ON questions(paper_id);
CREATE INDEX idx_questions_type ON questions(question_type);
CREATE INDEX idx_qopts_question ON question_options(question_id);
```

## 6. 字段映射（JSON -> DB）

### 6.1 顶层映射

| JSON路径 | 表字段 | 说明 |
|---|---|---|
| `subject_type` | `papers.subject_code` | 同时用于分科 |
| `papers[i]` | `papers` | 每个元素一行 |

### 6.2 试卷映射

| JSON路径 | 表字段 | 说明 |
|---|---|---|
| `papers[i].paper_title` | `papers.paper_title` | 试卷名 |
| `i+1`（导入顺序） | `papers.paper_no` | 科目内卷序号 |
| `papers[i].subject_type` | `papers.subject_code` | 应与顶层一致 |

### 6.3 题目映射

| JSON路径 | 表字段 | 说明 |
|---|---|---|
| `questions[j].question_id` | `questions.question_no` | 卷内题号 |
| `questions[j].question_type` | `questions.question_type` | 题型 |
| `questions[j].stem` | `questions.stem` | 题干 |

### 6.4 选项映射

| JSON路径 | 表字段 | 说明 |
|---|---|---|
| `questions[j].options` 的键 | `question_options.option_key` | A-F |
| `questions[j].options` 的值 | `question_options.option_text` | 选项文本 |

## 7. 导入顺序建议

1. 初始化 `subjects`
2. 导入 `papers`
3. 导入 `questions`
4. 导入 `question_options`

建议使用事务：

- 单文件一个事务（`math_1` 一次、`math_2` 一次、`math_3` 一次）
- 失败可回滚，保证一致性

## 8. 数据质量校验 SQL

### 8.1 每科试卷数量

```sql
SELECT subject_code, COUNT(*) AS paper_count
FROM papers
GROUP BY subject_code
ORDER BY subject_code;
```

### 8.2 每科题目数量

```sql
SELECT p.subject_code, COUNT(*) AS question_count
FROM questions q
JOIN papers p ON q.paper_id = p.id
GROUP BY p.subject_code
ORDER BY p.subject_code;
```

### 8.3 检查选择题是否有选项

```sql
SELECT q.id, p.subject_code, p.paper_no, q.question_no
FROM questions q
JOIN papers p ON q.paper_id = p.id
LEFT JOIN question_options o ON o.question_id = q.id
WHERE q.question_type = 'single_choice'
GROUP BY q.id, p.subject_code, p.paper_no, q.question_no
HAVING COUNT(o.id) = 0;
```

### 8.4 检查非选择题是否误写选项（可选）

```sql
SELECT q.id, p.subject_code, p.paper_no, q.question_no, q.question_type, COUNT(o.id) AS opt_cnt
FROM questions q
JOIN papers p ON q.paper_id = p.id
JOIN question_options o ON o.question_id = q.id
WHERE q.question_type <> 'single_choice'
GROUP BY q.id, p.subject_code, p.paper_no, q.question_no, q.question_type
HAVING COUNT(o.id) > 0;
```

## 9. 常用查询模板

### 9.1 查询数学二的所有填空题

```sql
SELECT p.paper_no, p.paper_title, q.question_no, q.stem
FROM questions q
JOIN papers p ON q.paper_id = p.id
WHERE p.subject_code = 'math_2'
  AND q.question_type = 'fill_blank'
ORDER BY p.paper_no, q.question_no;
```

### 9.2 查询某题及其选项

```sql
SELECT p.subject_code, p.paper_no, q.question_no, q.stem, o.option_key, o.option_text
FROM questions q
JOIN papers p ON q.paper_id = p.id
LEFT JOIN question_options o ON o.question_id = q.id
WHERE p.subject_code = 'math_1'
  AND p.paper_no = 1
  AND q.question_no = 3
ORDER BY o.option_key;
```

### 9.3 各科题型分布

```sql
SELECT p.subject_code, q.question_type, COUNT(*) AS cnt
FROM questions q
JOIN papers p ON q.paper_id = p.id
GROUP BY p.subject_code, q.question_type
ORDER BY p.subject_code, q.question_type;
```

## 10. 为什么这样设计（关键取舍）

### 10.1 不保留 `paper_id/source_question_id/crawled_at`

这是按你的“final简化策略”来定：目标是业务可用、结构清爽，不做来源追踪。

### 10.2 为什么仍保留 `paper_no`

因为去掉了来源 `paper_id` 后，仍需要一个稳定的卷定位键。  
`paper_no + subject_code` 可以稳定标识一套卷。

### 10.3 为什么 options 独立成表

独立成表比 JSON 文本字段更适合：

- 统计某选项频率
- 检索选项内容
- 数据一致性约束（A-F）
